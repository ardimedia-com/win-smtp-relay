## [Unreleased]

## [1.0.0-beta1-build33] - 2026-06-08

### Added

- Installer **"Reset the admin@local password" option** (on the setup Options dialog — also reachable when re-running the installer and choosing *Change*). When ticked, the installer sets a flag that makes the service, on its **next start**, regenerate the `admin@local` password exactly like a fresh install: a new one-time password written to `initial-admin-password.txt` and the Windows Event Log, flagged must-change-on-sign-in, and any account lockout cleared. This is the recovery path when the admin password is lost. The flag (a registry value the service can clear) is consumed once and cleared.

## [1.0.0-beta1-build32] - 2026-06-08

### Added

- The one-time **initial admin password** is now written to **`initial-admin-password.txt`** next to the service binaries on first run, and the **sign-in page links to it** (first run only), so you no longer have to dig through the Event Log. The file is **deleted automatically** the first time the password is changed (and the sign-in hint disappears). The password is still also logged to the Event Log as a fallback.

### Changed

- The admin-UI **firewall rule is now opened only when network access is enabled** during setup (the `NETWORKACCESS` option). With the default loopback-only install, no firewall rule is created for the admin port — secure by default. The SMTP firewall rules (25 / 587 / 465) are unchanged.
- The installer's final screen now points at `initial-admin-password.txt` for the initial sign-in credentials.

## [1.0.0-beta1-build31] - 2026-06-08

### Fixed

- Fixed the installer **aborting (error 1603)** at the port-detection step with "SFXCA: Failed to get requested CLR info (0x80131700)". The managed custom action lacked a CLR host config, so on machines with only .NET Framework 4.x it could not load. Added `CustomAction.config` (`supportedRuntime v4.0.30319` + `useLegacyV2RuntimeActivationPolicy`), and made the port-detection / config-write custom actions **non-fatal** (`Return=ignore`) so they can never abort the install — if they cannot run, the install completes with the default admin port (8025). No additional .NET Framework is required (4.8 ships with Windows 10/11); the action targets .NET Framework because WiX/DTF custom actions cannot run on .NET 10.

## [1.0.0-beta1-build30] - 2026-06-08

### Fixed

- Installer: a leftover `WinSmtpRelay` service from a previously failed or crashed install no longer blocks reinstallation. The installer now stops and deletes any existing service before installing it, so `ServiceInstall` no longer fails with "Service could not be installed. Verify that you have sufficient privileges" (error 1923). Note: a service already *marked for deletion* by an open handle (e.g. `services.msc`/Event Viewer open, or a crash-looping process) still requires a reboot — no installer can override that.

## [1.0.0-beta1-build29] - 2026-06-08

### Fixed

- Fixed the installed service **crashing at startup** with "Unable to configure HTTPS endpoint … the default developer certificate could not be found" when the self-signed admin certificate couldn't be prepared under the service account. The admin HTTPS endpoint no longer falls back to the ASP.NET Core developer certificate (which is absent on a server). The self-signed certificate is now loaded with a **cascade of key-storage strategies** (machine then user key sets) so it works under the restricted `NetworkService` account, and any failure is written to the **Windows Event Log**. If no certificate can be prepared, the service serves HTTP on loopback instead of crashing.

## [1.0.0-beta1-build28] - 2026-06-07

### Added

- Windows installer: a **Start menu** entry ("WIN-SMTP-RELAY") that opens the admin UI, and the installer's final screen now shows the admin URL.
- **Import an HTTPS certificate** for the admin UI from a new host-only page (Server → **HTTPS Certificate**): upload a PFX / PKCS#12 (with its password), which is validated (must include a private key), stored password-less and encrypted at rest (DPAPI), and applied to **new connections immediately — no service restart**. A "Remove" action reverts to the built-in self-signed certificate. Migration `AddAdminCertificateSettings`.
- Installer **port check**: on a fresh install the setup detects whether TCP **25** and **8025** are already in use. If 8025 is taken it automatically picks the next free port (8125, 8225, …) and configures the admin UI on it; if **25** is in use it shows a warning (port 25 can't be changed). The chosen admin port flows into the config, the Start-menu shortcut, the firewall rule, and the final-screen URL.
- Installer **network-access option**: a new setup dialog asks whether the admin UI should be reachable from the network. **Off by default** (binds to `127.0.0.1`, this computer only); when enabled it binds to `0.0.0.0` and the admin-port firewall rule is opened. The choice is written to `appsettings.Production.json`, layered over the shipped config.
- Setup checklist: a host-only **Admin UI HTTPS certificate** item under Basic — shows whether a trusted certificate is imported or the built-in self-signed one is in use, linking to the HTTPS Certificate page.

### Changed

- The admin UI is now served over **HTTPS by default**. When no certificate is configured, the service generates a persistent **self-signed certificate** (stored next to the service binaries) and binds HTTPS to it, instead of falling back to plain HTTP — so the management plane is always encrypted. Browsers show a one-time warning for the self-signed certificate; a configured PFX (`AdminUi:CertificatePath`) still takes priority, and a certificate can be imported at runtime from the admin UI (see Added).

## [1.0.0-beta1-build27] - 2026-06-07

### Added

- Web-admin authentication: sign-in is now required for the Blazor admin UI and the REST API. Built on ASP.NET Core Identity with a tenant-aware admin account model and the roles `HostAdmin`, `TenantAdmin`, and `TenantViewer`.
- Dual authentication schemes: interactive cookie login for browsers and per-tenant hashed API keys (`X-Api-Key` header or `Authorization: Bearer`) for automation. Both produce the same role/tenant claims.
- Authorization policies `AdminView`, `AdminFull`, and `HostAdmin`: read endpoints require `AdminView`, mutating (non-GET) endpoints require `AdminFull`; `/api/health` remains anonymous.
- Initial host administrator (`admin@local`) is seeded on first run with a one-time random password written to the log (Event Log + console); the account must change its password on first sign-in.
- `Tenant` entity and a seeded default tenant; admin users and API keys are tenant-bound (foundation for multi-tenancy).
- Login, change-password, and access-denied pages plus a sign-out endpoint.
- Show/hide (eye) toggle on every password field across the account pages (sign-in, change password, reset password, sign-up), so the entered value can be revealed to confirm it.
- Forgot-password / reset-password flow: a "Forgot your password?" link on the sign-in page emails a reset link (via the relay's own delivery pipeline, when a system from-address is configured) with a neutral, no-enumeration response; the reset page sets a new password via the Identity reset token. The sign-up page gained a "Sign in" link back to login.
- Tenant management: host administrators can create, enable/disable, and delete tenants from a new host-only **Tenants** admin page and `/api/tenants` (gated by the `HostAdmin` policy). The default tenant cannot be deleted. Deleting a tenant now permanently purges all of its data — relay users, mail queue, delivery logs, connectors, domains, DKIM keys, message filters, statistics, API keys, and admin accounts — in a single transaction (the UI requires an explicit confirmation).
- Health page: for each sender domain it shows the required SPF / DKIM / DMARC records versus the live published DNS, with a match / missing / mismatch status, copy buttons, an ad-hoc domain check, and re-check. The expected DKIM record is derived from the configured signing key; recommended SPF/DMARC values come from a new `Dns` configuration section (sending IPs, hostname, includes, DMARC policy/report address). Read-only guidance — the relay never modifies DNS.
- Health page now verifies the full deliverability picture, not just sender SPF/DKIM/DMARC, grouped into **Domains** and **IP addresses**: MX records for recipient/backup-MX domains (must point to this relay), the public hostname's A/AAAA record (must resolve to a sending IP), reverse DNS (PTR) for each sending and per-tenant egress IP with FCrDNS hostname matching, an SPF-coverage check that flags a tenant egress IP not listed in the sending IPs, and a **DNS blocklist (DNSBL)** check of each sending IP against Spamhaus ZEN and SpamCop (a listing is why Microsoft 365 and others return SMTP 5.7.1; the check links to delisting and distinguishes a real listing from a resolver that refused the query). All live-checked via DnsClient and read-only.
- Self-service signup can now be configured at runtime from the admin UI: a host-only **Settings** page (`/settings`) has a switch to enable/disable signup and a field for the verification/reset email **from-address**, both applied immediately with no restart. The values live in the database (a new `PortalSettings` row); the enabled flag is seeded once from the appsettings `AdminUi:SelfServiceSignupEnabled` value, and the from-address falls back to `AdminUi:SignupFromAddress` when left blank. The sign-in, sign-up, and forgot-password pages read these live.
- Self-service signup is rate-limited per client IP (`AdminUi:SignupMaxAttemptsPerIpPerHour`, default 5; 0 disables) to limit abuse. Over-limit attempts are rejected with a neutral message before any tenant or account is created. The limit is editable at runtime on the **Settings** page (stored in `PortalSettings`, seeded from appsettings).
- Self-service tenant signup (opt-in, `AdminUi:SelfServiceSignupEnabled`, off by default): an anonymous `/signup` page creates a **disabled** tenant plus a locked, unverified Tenant Admin. The account activates either by clicking an email verification link (sent through the relay's own delivery pipeline when `AdminUi:SignupFromAddress` is set) or by host approval — enabling a pending tenant on the Tenants page confirms and unlocks its admins. A "Create an organization" link appears on the sign-in page when signup is enabled.
- Admin password reset: host/tenant administrators can reset another admin's password from the **Admin Users** page. A new one-time password is shown once, the account is flagged to change it on next sign-in, and the reset rolls the user's security stamp so any existing sessions are invalidated. This is the recovery path when self-service email reset is unavailable.
- Admin-user management UI: an **Admin Users** page (gated by `AdminFull`) to create, enable/disable, and delete web-admin login accounts. The scope follows the tenant switcher — in a tenant scope you manage that tenant's admins (Tenant Admin/Viewer roles); in "All tenants" you manage host administrators. New accounts get a one-time password and must change it on first sign-in; you cannot disable or delete your own account.
- API-key management UI: create, list, and revoke API keys from a new **API Keys** page and `/api/apikeys` (both gated by `AdminFull`). Keys are scoped to the current tenant (the host admin's switcher selection); the secret is shown only once on creation, and tenant-scoped admins cannot mint host-level keys.
- Tenant switcher: a header control (host admins only) selects the tenant to work within ("All tenants" or a specific tenant). The selection — carried as an `active_tenant` claim and honored by both the request middleware and the Blazor circuit — scopes every per-tenant admin page and API call to that tenant. Blazor pages were moved to a tenant-propagating scope factory so their per-operation database scopes inherit the active tenant.
- `TenantId` on all per-tenant entities (relay users, receive/send connectors, accepted/sender domains, IP access rules, domain routes, DKIM domains, message-filter rules, queued messages, delivery logs) with a foreign key to `Tenant`; existing rows are assigned to the default tenant. Usernames are unique per tenant (composite index); accepted recipient and sender domains are globally unique (one tenant per domain — see Security); DKIM domains remain per-tenant. `DailyStatistics` is also tenant-owned (composite `TenantId`+`Date` key); `RateLimitSettings` is intentionally a single host-level row (the per-IP/sender/auth limits protect the shared listener).
- Domain ownership verification for **both** accepted sender and recipient domains: each domain gets a random token and shows the DNS TXT record (`winsmtprelay-verification=<token>`) to publish at the domain; a "Verify" action on the Accepted Domains (Sender) and Accepted Domains (Recipient) pages performs a live DNS TXT lookup and marks the domain verified (Verified/Unverified status).
- Optional **verification enforcement** (off by default), on the Settings → Email auth tab: "Require verified ownership for accepted sender domains" gates the SMTP `MAIL FROM` path, and "Require verified ownership for accepted recipient domains" gates the `RCPT TO` path — an unverified-but-configured domain is rejected. Applies only among configured domains (an empty list still means "accept all"); backup-MX domains are unaffected. The verified-domain sets are cached on the SMTP hot path and refreshed when a domain is verified or the toggle changes.
- Optional **strict tenant binding for unauthenticated submission** (off by default), on the Settings → Email auth tab. By default an unauthenticated message's tenant is guessed from the MAIL FROM domain (falling back to the default tenant). "Bind the tenant to the matching allow IP rule" instead derives the owning tenant from the tenant's own Allow IP rule and **rejects a MAIL FROM domain owned by a different tenant**, closing cross-tenant sender-domain spoofing (ambiguous IP matches are rejected). A client permitted only by a shared host-baseline allow rule isn't bound to any tenant — it may send only as an unclaimed or host-owned domain, otherwise it's rejected, so each tenant needs its own allow rule (or SMTP AUTH). "Reject when no tenant can be determined" refuses unattributable mail instead of silently using the default tenant. Authenticated sessions are unaffected — their tenant comes from the SMTP user.
- Per-tenant outbound source IP (egress IP): a tenant can be assigned a source IP on the Tenants page, and outbound SMTP delivery for that tenant's mail binds its local socket to that address (the host must own the IP; unset uses the OS default). Useful for per-tenant IP reputation/SPF separation.
- Per-tenant delivery statistics: daily statistics are aggregated and stored per tenant, and the monthly view sums only the rows visible in the current scope — a tenant administrator sees its own totals, while a host administrator with "All tenants" selected sees the host-wide total. (The live/hourly/daily charts already scoped correctly, since they read the tenant-filtered delivery logs directly.)
- Tenant-scoped data access: admin requests (cookie or API key) and Blazor circuits are scoped to the authenticated principal's tenant via an ambient `ICurrentTenant`, enforced by an EF Core query filter on every tenant-owned entity. Host administrators see all tenants; tenant administrators see only their own tenant, and rows they create are stamped with their tenant on insert. Background services (delivery, statistics, config cache) run unscoped across all tenants.
- **Setup** page (`/setup`): a live, per-tenant onboarding checklist that reflects the tenant's actual configuration. It shows two readiness tiers — *Can send mail* (the hard minimum: an active tenant plus a way for clients to submit — SMTP credentials or an allowed-IP rule) and *Deliverability* (recommended best practice) — a "next best action" banner pointing at the most important incomplete step, and the setup items grouped into **Basic** (the organization and the sending identity it relays with — active tenant, public hostname + IP), **Recommended** (secure submission via SMTP credentials, plus deliverability best practice), and **Optional** with live Done / In-progress / To-do / Blocked / Default status badges and one-click links to each configuration page. Intentional permissive defaults (empty domain/IP lists meaning "allow all") read as *Default* rather than as errors. The page also runs a live SPF/DKIM/DMARC check (after first paint, so the checklist appears instantly) and, when records are missing or mismatched, shows a **DNS records to publish** card with the exact expected values and copy buttons. A **Send a test message** action (restricted to `AdminFull` admins, server-side, and to the tenant's configured sender domains) queues a message through the relay's own delivery pipeline so the operator can confirm outbound delivery (MX lookup, TLS, DKIM) works end-to-end and watch the result under Journal. Linked first under **Monitor** in the sidebar. Host admins in "All tenants" scope are prompted to pick a tenant via the switcher.

- Journal page: delivery-log rows are now clickable and open a detail dialog showing the full status message (e.g. the complete 5xx delivery error), the remote server, and the message/queue id, so an error truncated in the table can be read in full. The "Time" column header (previously "Time (UTC)") was corrected now that the value is shown in the viewer's local time zone.
- The Settings **Sending identity** tab can now auto-detect the relay's public outbound IP: a **Detect public IP** button does a best-effort external lookup (via `api.ipify.org` over `IHttpClientFactory`) and offers a one-click **Add** for each detected address next to **Sending IP addresses**. It is labelled as a proposal — confirm it's the IP your outbound mail actually uses.
- DKIM keys are now generated and stored entirely by the relay: creating a DKIM domain on the **DKIM Domains** page generates an RSA key pair server-side and stores the private key (PEM) in the database, then shows the public-key DNS TXT record to publish — no operator-supplied key file or host filesystem path is required. A **Regenerate** action rolls the key and shows the new record. The previous private-key file-path option remains supported as a legacy fallback for installs that manage keys on disk. Migration `AddDkimPrivateKeyPem` included.
- The **Health** page now shows an **Outbound IP** check that detects this server's current public IP (best-effort, via the same external lookup as Settings) and flags a **mismatch** when it is not among the configured sending IPs — so a changed dynamic ISP address is surfaced instead of the per-IP checks silently validating a stale, no-longer-current address. On a mismatch it links straight to Settings → Sending identity.
- Per-tenant **suppression list**: recipient addresses that hit a permanent (5xx) bounce — or are added manually — are recorded per tenant, and the delivery worker refuses to deliver to a suppressed address (it is skipped and logged; a message whose recipients are all suppressed becomes `Suppressed` and is not retried). A new **Suppression List** admin page lists entries and lets you add or remove addresses. Repeatedly mailing dead addresses is a primary cause of blocklisting, so this protects sending reputation. Migration `AddSuppressionList` included.
- **Email reporting** (host-level, off by default): a background service sends a **daily activity digest** — volume delivered/bounced/deferred/suppressed over the last 24h, bounce rate, queue depth, suppression-list size, and the DNSBL status of each sending IP — plus **immediate incident alerts** when a sending IP becomes blocklisted or the 24-hour bounce rate exceeds a configurable threshold. Sent through the relay's own delivery pipeline and configured on a new **Reporting** tab on the Settings page (recipient, optional from-address, daily digest time, bounce-rate threshold). Migration `AddReportingSettings` included.
- One-click **Brevo** smart-host setup. The Send Connectors page and the Setup → Basic checklist now show a short explainer — why you might relay through Brevo (managed sending IPs/reputation, inbox deliverability without running your own outbound MX), the free **300 emails/day** tier, the free account + **SMTP key** you need, and links to brevo.com — plus an **Add Brevo Connector** button. The button opens the New Send Connector form pre-filled for Brevo (`smtp-relay.brevo.com:587`, TLS required via `?template=brevo`); the admin only supplies their Brevo account email (username) and SMTP key (password, now stored encrypted). Setting the connector as **Default** routes all of the tenant's outbound mail through Brevo.

- **Data retention** (host-level, Settings → Data retention): configurable, automatically-enforced retention for every category of accumulating data, replacing the old statistics-only retention tab. Per-category windows for **message history** (terminal queued messages), the **delivery audit log** (Journal), **statistics**, and the **suppression list**, plus a **strip message body on delivery** toggle (drops the raw email content once delivered, keeping only metadata). Selectable best-practice **profiles** fill the values — *Standard* (GDPR/FADP-balanced; the default), *Privacy-first* (minimal), *Financial* (MiFID II / SEC-FINRA, ~7-year content retention), and *Healthcare* (HIPAA, ~6-year) — each overridable to *Custom*. The delivery-audit-log window has a hard minimum floor (7 days) so delivery evidence can never be scrubbed by a configuration change, and the page warns when bodies are kept long-term or a regulated profile is selected (the relay sets retention durations but is not a certified WORM/encrypted archive). All purges run in the existing nightly maintenance pass across all tenants. Profiles are starting points, not legal advice. Seeded from a new appsettings `DataRetention` section; migration `AddDataRetentionSettings` included.

### Changed

- The browser tab title is now consistently **WIN-SMTP-RELAY** on every page. Interactive pages previously showed the URL (e.g. `localhost:8025/update`) because no `PageTitle` was rendered, so the interactive `HeadOutlet` blanked the document title on navigation; an always-on `PageTitle` in the main layout fixes it.
- Fixed the Deliverability hostname (A/AAAA) and reverse-DNS (FCrDNS) checks intermittently reporting a resolvable hostname as "not found" / "FCrDNS incomplete". The address lookup awaited the A and AAAA queries inside one try/catch, so a transient AAAA failure (most hosts have no IPv6 record, and a momentary resolver hiccup throws) discarded the already-resolved A result. A and AAAA now resolve independently — a failed or empty AAAA can never hide a present A. The public-resolver per-query timeout was also raised back to 5s (with 2 retries) so a cold/slow-but-alive `8.8.8.8` response isn't mistaken for a failure (which would route to the flaky secondary).
- The Deliverability page no longer appears to hang while loading. Three changes: (1) its live DNS checks now run **concurrently** instead of sequentially — it reads the configuration once, then fans out all the SPF/DKIM/DMARC/MX/hostname/PTR/blocklist lookups in parallel (each in its own scope), so total wait is the slowest single lookup rather than the sum of all of them; (2) a **loading indicator** with a live "checking N of M" progress count replaces the blank skeleton; (3) the public DNS resolver no longer picks between Google (`8.8.8.8`) and Cloudflare (`1.1.1.1`) at random per query — DnsClient's random server selection meant that on a network where one resolver is slow or blocked, roughly half the lookups burned the full timeout. The order is now fixed (always `8.8.8.8` first, `1.1.1.1` only as fallback) with a tightened 2.5s per-query timeout.
- Delivered message bodies are now dropped on delivery by default (data minimisation), and the message queue history, delivery audit log, and suppression list are now subject to retention windows. Previously delivered messages — including their full raw body — and delivery-log rows were kept indefinitely; only aggregated statistics were ever purged. A message's `CompletedUtc` is now also stamped for the Failed and Suppressed terminal states (previously only Delivered/Bounced).
- Setup checklist statuses are simplified to **Done / In progress / To do**. The former "Default" (permissive) and "Blocked" states are gone: a working default (direct-to-MX, no IP restrictions, shared sending IP, verification not enforced, etc.) now reads as **Done** instead of a separate "Default" tag; a genuine gap (no sender domains, no DKIM) is a plain **To do**; and a former "Blocked" prerequisite (e.g. "add a sender domain first") is shown as an ordinary To-do task rather than an alarming "Blocked".
- The Setup checklist now validates the **sending identity live**, not just that the fields are filled in: the *Sending identity* row drops to **In progress** (linking to the Deliverability page) when the public hostname doesn't resolve to a configured sending IP, or the detected outbound IP isn't among your sending IPs — surfacing split-horizon / wrong-IP problems on the checklist itself instead of only on the Deliverability page.
- The Setup → Basic section now presents the two ways to send outbound mail side by side under "Sending outbound mail — pick one": **Relay through Brevo** (a smart host) and **…or send directly from your own IP** (host admins only; links to the tenant's Outbound (sending) IP + the Test-outbound check). This is the actual fix path when the Outbound tile shows "delivery failing" — adding sender domains does not affect delivery.
- The Setup **next best action** banner now prioritizes outbound delivery: when the Outbound tile is *failing* (or *unreliable*) it points at "Set up outbound sending" (with a one-click **Set up Brevo** action) instead of suggesting a best-practice step like sender domains — since a delivery failure is the real blocker, and sender domains do not fix it.
- The **Relay through Brevo** card (Send Connectors page and Setup → Basic) now ends with a highlighted "Why prefer this?" note explaining the deliverability advantage of relaying through warmed, monitored IPs over sending directly from your own IP (which needs a static IP, correct reverse DNS, and a clean reputation).
- Setup readiness tiles now separate **Inbound (submission)** — *"Relay can receive mail"*, i.e. clients can submit to the relay — from **Outbound (delivery)** — *"Can send mail"*, driven by the actual recent delivery outcomes (healthy / some failing / failing / not tested yet). A green "can send" therefore no longer hides failing outbound delivery. The deliverability tier is renamed **Best practices**, and the **Basic** and **Best practices** group headings show an *N of M in place* completion count.
- The per-tenant outbound source IP is relabelled **Outbound (sending) IP** (previously "Egress IP") across the Tenants pages and the Deliverability page. The tenant editor can now **detect the host's local IPv4 addresses** (click one to fill the field) and **test outbound** SMTP from the chosen source IP — port 25 to a public MX (direct delivery) and port 587 to Brevo (smart host) — so you can confirm that IP actually reaches the internet on the SMTP ports before relying on it. The target hosts are fixed, so the test cannot probe arbitrary addresses.
- The **Setup** page was refined from a deep review: added a **Suppression list** item; moved the **Send a test message** card below the readiness summary and next-action banner (it's a post-setup verification, not the first thing you see); the **In progress** status now has its own badge colour distinct from **To do**; the DNS-record copy buttons got an accessible label; and the **About** dialog gained a "Receiving mail" note (inbound needs a host SMTP listener plus an MX record pointing at it) and a short SPF / DKIM / DMARC / MX / TLS glossary.
- "Delivered"/success states now use a dedicated semantic **`--success`** green token instead of the data-viz `--chart-2` (which is red-orange in the default theme, nearly indistinguishable from errors in the Live Activity feed) and the achromatic `--primary` (grey) for the success badge. The light-mode `--warning` colour was also darkened so warning text meets AA contrast on its tinted background.
- The browser tab title is now always **WIN-SMTP-RELAY** on every page (previously "WIN-SMTP-RELAY Admin", with per-page suffixes like "Sign in — WIN-SMTP-RELAY" on the account pages). The per-page `<PageTitle>` overrides were removed.
- The left navigation now follows a clear **"All tenants = monitor & operate; pick a tenant = configure"** model, driven by the host-only tenant switcher. The **All tenants** overview shows only the consolidated, cross-tenant views — Dashboard, Live Activity, Queue, Journal, Admin Users (host administrators), Tenants, Event Log, Update. Selecting a specific tenant (e.g. the default tenant) reveals that tenant's configuration: Setup, Deliverability, the **Configuration** group (Send Connectors, Accepted Domains, IP Access Rules, DKIM, Message Filters, SMTP Users, Suppression List), API Keys, and — for a host admin only — a **Server** group with the server-wide pages (Settings, Receive Connectors, Rate Limits, Startup Config). The server-wide pages apply to the whole server regardless of which tenant is selected. Tenant admins and viewers are always scoped to their own tenant and never see the host-only Server/Host items (enforced by both scope and the `HostAdmin` role; the pages are independently `HostAdmin`-gated at their routes). Account management requires `AdminFull`; DKIM is host-admin only.
- Detail and settings forms now use the full content width instead of a narrow fixed cap, with multi-field forms laid out in a responsive column grid (Rate Limits, the message-filter rule editors, domain routes, admin-user/API-key/tenant pages, the send/receive connector editors, IP access rules, and SMTP users). Intentionally narrow single fields (e.g. the tenant egress IP) are kept.
- The **Settings** page now groups every tab's fields into bordered sub-panels (matching the Sending identity tab's Basic/SPF/DMARC panels): Email auth (Inbound checks / Domain ownership / Tenant binding), Signup, Backup MX, and Statistics.
- The **Event Log** page (`/logs`) now renders immediately with a loading indicator and reads the Windows Event Log after first paint on a background thread, instead of blocking the initial render on a slow synchronous read.
- All admin lists now follow a consistent list/detail pattern: a page opens to just the list with a **New** button, clicking a row opens a dedicated detail page (not a pop-up dialog) for editing, creating an item also opens a detail page, and per-row actions (edit, delete, retry) are shown as inline icons instead of dropdown menus or inline forms. Applies to Receive Connectors, Send Connectors, Domain Routes, DKIM Domains, IP Access Rules, Users, Admin Users, Tenants, Accepted Domains (recipient and sender), API Keys, Message Filters (header and sender rewrites), and the Message Queue. Tabbed pages (Send Connectors, Message Filters) restore the active tab via a `?tab=<key>` query parameter after returning from a detail page.
- The **Settings** page is now organised into tabs (Sending identity, Email auth, Signup, Backup MX, Statistics) ordered by importance, each deep-linkable via `?tab=<key>` — so the Setup checklist's "Sending identity" item links straight to the DNS tab. The Sending identity tab's fields are visually grouped into **Basic** (public hostname, sending IPs), **SPF** (includes, "all" qualifier), and **DMARC** (report address, policy, percentage) sub-sections.
- The **Health** page is now organised into **IP addresses** (default — public hostname A/AAAA, reverse DNS, blocklist per sending/egress IP) and **Check DNS** tabs, deep-linkable via `?tab=ip|dns`. The Check DNS tab adds a **"Used in your configuration"** panel that offers the domains found in your settings as one-click checks — the public hostname's registrable domain (e.g. `ardimedia.com` from `smtp2.ardimedia.com`, resolved via the **Public Suffix List** so multi-level TLDs like `co.uk` are handled correctly), DKIM domains, and the DMARC report-address domain — alongside the ad-hoc domain box (now with inline validation) and a link from each configured IP back to the IP tab. Ad-hoc/candidate results persist across "Re-check all".
- When a domain already publishes an SPF record that doesn't authorise this relay, the Health page now shows a **"To publish" merged record** — the published record with the relay's missing mechanisms inserted before the `all` qualifier — so the operator copies one complete record instead of replacing (and breaking) their existing one. SPF permits only a single record per domain; the merge keeps the published senders and `all` qualifier and adds only what's missing.
- The SPF check now warns when a record exceeds the **RFC 7208 limit of 10 DNS-querying terms** (`a`, `mx`, `ptr`, `exists`, `include`, `redirect`), following nested `include:`/`redirect=` targets to estimate the true count — an over-limit record fails with a permerror, so receivers treat it as if no SPF were published. The warning is computed for whichever record the operator would publish (the merged "To publish" value, the recommended record, or the published one).
- The Setup page's **Basic** group now shows a "can't send mail yet" notice when the organization is active but has no submission path (no SMTP users and no allowed client IPs). Submission access lives under Recommended, so Basic could otherwise read all-green while the *Can send mail* tile was red; the notice links to add an SMTP user or an IP access rule.
- The product name shown to users is now **WIN-SMTP-RELAY** across the admin UI and the emails it sends (page titles, sidebar/header, help text, verification/reset/test messages). Technical identifiers are unchanged — namespaces/assemblies, the `WinSmtpRelay` Windows service and Event Log source names, the auth cookie name, and the SMTP server banner all stay `WinSmtpRelay`.
- The page heading is again focused on navigation for screen readers (`FocusOnNavigate Selector="h1"`); the focus outline that previously appeared on the `<h1>` is now suppressed in CSS instead of by disabling focus management.
- Form accessibility: every form label across the admin pages is now programmatically associated with its control (`<label for>` ↔ matching `id`), so clicking a label focuses its field and screen readers announce the label/control pairing. Covers ~95 fields across 19 pages — text inputs, Bb selects, and the native enforcement/role/SPF/DMARC dropdowns. The Settings toggle switches, which are described by adjacent text rather than a `<label>`, were given explicit accessible names (`aria-label`).
- The DNS/deliverability page is renamed **Health → Deliverability** (sidebar label and icon, page heading, breadcrumb), since it now verifies the whole delivery picture — SPF/DKIM/DMARC, MX, reverse DNS, blocklists, and the outbound IP — not just a health snapshot. The page also responds at `/deliverability`; the old `/health` URL still works and internal links were updated. The **Startup Config** sidebar item got its own icon (it previously shared the Settings gear).
- The **Email Volume** chart now draws **Sent** in the semantic success green and **Failed** in the destructive red, instead of the theme-dependent data-viz `--chart-1`/`--chart-2`, so the legend key matches the green = delivered / red = failed convention used in the Live Activity feed and status badges across every theme.
- **Suppression List** rows are now clickable and open a read-only detail dialog showing the full bounce/complaint detail (which is truncated in the table), with **Remove** available from the dialog — matching the Journal row-click pattern.
- Sign-up page: clearer wording (consistently "organization" rather than "tenant") and a hint under the **Identifier (slug)** field explaining its format and that it cannot be changed later.
- Detail-page headings are now consistent — an existing record reads `<Noun>: <value>` (e.g. "Send connector: relay-out", "IP rule: 10.0.0.0/8", "SMTP user: app1") instead of a mix of "Edit: <value>" and "<Noun>: <value>"; the DKIM list page is titled **DKIM Domains** to match the other list pages.
- Internal: the RFC822 system-mail composer (verification, password reset, the Setup test message, and the reporting digest/alerts) is now a single `SystemEmail` helper in the core library, replacing two byte-identical copies so the CRLF header-injection sanitization and header set can never diverge between callers.
- All persisted/exposed timestamps moved from `DateTime` (UTC by convention) to `DateTimeOffset`, so the type is offset-aware end to end (entities, services, API DTOs, SignalR payloads). Because SQLite has no native `DateTimeOffset`, a model convention stores each value as a fixed-width UTC ISO-8601 string (`...Z`) — human-readable and offset-marked in the database, and (being fixed width) orderable and range-filterable by SQLite. Date-only/time-of-day values stay `DateOnly`/`TimeOnly`; genuinely transient/internal `DateTime` (scheduling math, process uptime, the Windows Event Log viewer) is unchanged. List/recent queries now order by the autoincrement `Id` (chronological by construction) since SQLite can't `ORDER BY` the column directly, and the delivery queue's eligibility filter runs in memory over the (small) queued set. Migration `ChangeTimestampsToDateTimeOffset` included.
- Timestamps in the admin UI are now shown in the **viewer's local time zone** instead of UTC. Times are still stored in the database as UTC; a new per-circuit `BrowserTimeService` detects the browser's time zone (IANA id, with a fixed-offset fallback) and a `<LocalTime>` component converts on display, refreshing once the zone is known. Covers Dashboard (including the Email Volume chart's time axis), Queue, Journal, API Keys, Users, Tenants, accepted/sender domains, the Update page, and the live Activity feed; each value carries a tooltip showing the original UTC. (The Windows Event Log viewer keeps the server machine's local time, which is inherent to the Event Log.)
- The DNS recommendation inputs (public hostname, sending IPs, SPF includes/qualifier, DMARC report address/policy/percentage) that drive the **Health** page's recommended SPF/DMARC records are now runtime settings on the **Settings** page, applied without a restart. Stored in `DnsSettings` (lists as delimited strings), seeded once from the appsettings `Dns` section; the Health page reads them on its next check.
- Statistics retention (how many days of daily statistics to keep) and the daily aggregation time are now runtime settings on the **Settings** page; the background aggregator reads them each cycle, so changes apply without a restart. Stored in `StatisticsRetentionSettings`, seeded once from the appsettings `Statistics` section.
- Backup-MX behaviour (enabled, the held domains, retry interval, and max hold time) is now a runtime setting on the **Settings** page, applied immediately with no restart. It is stored in the database (`BackupMxSettings`, domains as a delimited list), cached on the SMTP/delivery path, and seeded once from the appsettings `BackupMx` section.
- Inbound email authentication (SPF/DMARC enabled + the failure enforcement mode: log-only / reject / quarantine) is now a runtime setting on the **Settings** page, applied immediately with no restart. It is stored in the database (`EmailAuthSettings`), cached on the SMTP path, and seeded once from the appsettings `EmailAuthentication` section.
- Rate limits (per-IP connections, per-sender messages, failed-auth auto-ban) are now read live from the database, so edits on the Rate Limits admin page take effect immediately. The limiter previously read the static appsettings values and ignored the database. The appsettings `RateLimit` section now only provides the initial values, applied until the settings are first edited in the UI. Per-user limits remain on the relay user.
- The SMTP listener now binds its endpoints from the enabled, host-level receive connectors stored in the database (the source of truth once seeded from appsettings), instead of reading the appsettings endpoints directly. Edits made on the Receive Connectors page therefore take effect on the next service restart; previously they had no effect. If the database has no enabled connectors, the listener falls back to the appsettings endpoints. Receive connectors define the host's shared listening sockets, so they are not per-tenant — the Receive Connectors page and `/api/connectors/receive` are now restricted to host administrators.
- The admin UI now binds to loopback (`127.0.0.1`) by default and serves over HTTPS (the ASP.NET Core development certificate in Development; configurable via `AdminUi:CertificatePath` / `AdminUi:CertificatePassword` for production). It previously bound `0.0.0.0` over plain HTTP.
- IP access rules stored in the database are now the authoritative source for relay IP authorization, evaluated with first-match Allow/Deny semantics by sort order. `Deny` rules are now honored (previously they were never evaluated), and edits made in the admin UI take effect immediately (cache invalidation). The static `SmtpListener:AllowedNetworks` list is used only as a fallback when no database rules exist.

### Fixed

- Button labels no longer wrap onto two lines (e.g. the Setup **Re-check** button rendering as "Re-" / "check"); applied globally to all buttons.
- The **Live Activity** feed — and the live updates on the Dashboard and Queue pages — now actually work. They previously opened a server-to-self SignalR connection to the authenticated `/hubs/activity`, but a connection initiated from the Blazor Server circuit carries no auth cookie or tenant, so it was rejected and sat permanently **Disconnected** (and the dashboard/queue never refreshed live). The server-rendered pages now subscribe to an **in-process activity feed** (`IActivityFeed`) instead — filtered to the circuit's tenant scope (a host admin in "All tenants" sees everything) — so events stream live with no SignalR round-trip. The Live Activity status badge now reads **Live**.
- The Deliverability page's public-record checks — public hostname (A/AAAA), MX, reverse DNS (PTR), and the SPF/DKIM/DMARC TXT lookups — now query a **public resolver** (8.8.8.8 / 1.1.1.1) instead of the host's own resolver, so they reflect what the internet actually sees rather than a split-horizon internal answer (e.g. a public hostname that resolves to a private 10.x address on the LAN). The DNS blocklist (DNSBL) check deliberately stays on the host's own resolver, since Spamhaus and similar refuse queries from large public resolvers.
- A send connector flagged **Default** is now actually used as the outbound routing fallback for a tenant's domains that have no explicit Domain Route (evaluated after domain routes, before the appsettings global smart host). The `IsDefault` flag was editable in the UI but ignored by the delivery path, so marking a connector "Default" had no effect and mail fell through to direct-MX delivery. (A default connector with no smart host still falls through to direct MX, as before.)
- Corrected help/empty-state text that described the opposite of the actual behaviour: the IP Access Rules empty state (it falls back to `AllowedNetworks` and denies others — it does not accept all), the Accepted Domains "About" (the empty-list default is *not* an open relay — external relay still needs auth or an explicit allow-IP rule), and the Journal (times are shown in the local zone, not UTC; the audit trail is per-recipient, not per-message).
- An invalid sender-rewrite **From Pattern regex** is now rejected at save with an error, instead of saving silently and then throwing at delivery time (which broke that tenant's mail flow with no feedback). The header-rewrite "Match Value" field is relabelled "exact match" — it is compared exactly, not as a regex.
- A delivery interrupted by a **service shutdown** is now requeued cleanly instead of being recorded as a failed attempt and bounced/retried. The per-MX delivery loop previously swallowed the shutdown's `OperationCanceledException` and re-wrapped it as a `DeliveryException` (status 421), so the delivery worker's clean-shutdown handling never triggered; the loop now propagates the cancellation when the service is stopping.
- A delivery **timeout** now reads "the connection timed out after Ns" in the Journal and Event Log, instead of the opaque raw exception text "A task was canceled".
- The email-reporting digest no longer **re-counts a suppressed recipient on every retry** of a mixed message — a suppressed-skip is logged once (on the first attempt), so the "Suppressed" tally is no longer inflated by `retries × suppressed`.
- The `badge-info` (Update page "update available") and `badge-destructive` (IP-rules "Deny") badge variants had no CSS and rendered as a plain uncoloured badge; they now have their own styling (info = blue, destructive = red), in both light and dark themes.
- Dark mode lost when switching tenants (and on other enhanced navigations): Blazor enhanced navigation patches the DOM to match the server response, which has no theme attributes, so it stripped the client-applied `dark` class and `data-theme`. The stored per-browser theme is now re-applied on Blazor's `enhancedload` event so it survives enhanced navigation and the tenant switcher's form post.
- Native `<select>` dropdowns (e.g. the host tenant switcher) were unreadable in dark mode — the OS-drawn option popup stayed light with near-white text. Added `color-scheme` to the light/dark roots so the browser paints native controls (option popups, scrollbars, carets) in the matching scheme, and pinned option colours to the themed popover.
- Warning/amber UI (status badges, the Setup "to-do"/can't-send indicators, dashboard warning values) was unreadable in the Amber and Storm dark themes: it reused the data-viz `--chart-3`, which is near-black there. A dedicated `--warning` colour (light amber in dark themes) is now used, fixing contrast without recolouring charts.
- Rate-limit settings edited in the admin UI had no effect and were silently reverted: the rate limiter read the appsettings values rather than the database, and the configuration seeder overwrote the database row with appsettings on every startup. Edits are now honored and persist (the seeder only applies appsettings to an untouched row).
- The SMTP client IP was always `null` because the listener read the wrong `ISessionContext` property key, which silently disabled the IP allow-list relay restriction, per-IP rate limiting, failed-auth auto-ban, and SPF source-IP checks (SPF fell back to loopback). The listener now uses the SmtpServer `EndpointListener.RemoteEndPointKey` constant at all call sites.
- Duplicate SMTP endpoint binding and a doubled `AllowedNetworks` list: pre-initialized collections in `SmtpListenerOptions` were appended to (not replaced) by configuration binding, producing two `0.0.0.0:25` endpoints and eight allowed networks from four. Bound collections now start empty.

### Security

- **DKIM private keys are now encrypted at rest** (Windows DPAPI, machine scope) instead of stored as plaintext PEM. Existing plaintext keys keep working and are re-encrypted on next save. The REST `GET /dkim/domains` no longer returns key material either; it projects to a summary DTO (`HasPrivateKey` flag instead of the PEM). The Blazor pages read the decrypted key transparently via the value converter.
- **Upstream smart-host / submission passwords are now encrypted at rest too** (the same DPAPI protector as the DKIM key, generalised to a shared `SecretProtector`). The `SendConnector.EncryptedPassword` field — for example a Brevo SMTP key entered when relaying through a smart host — was previously stored as **plaintext** in the SQLite database despite its name; it is now DPAPI-protected, with the same transparent converter, legacy-plaintext fallback, and no-migration (TEXT column) behaviour as the DKIM key. The REST `GET /api/connectors/send` and `GET /api/routes` no longer emit the password either — both project to summary DTOs (a `HasPassword` flag instead of the secret), so the credential never leaves the server over the API. The Blazor connector editor still reads the decrypted value via the service for an authorised admin over HTTPS.
- **"Require TLS" on a send connector is now enforced** instead of being a stored-but-ignored control. The delivery path previously honoured only "Opportunistic TLS"; a connector (or the global smart host) marked Require TLS now uses **mandatory STARTTLS** and fails the delivery if the server will not negotiate TLS, rather than silently falling back to an unencrypted connection. Direct-to-MX delivery deliberately stays opportunistic so it can still reach MX hosts that offer no TLS.
- **Web-admin write pages are now gated to the `AdminFull` role, matching the API.** The per-tenant configuration pages (SMTP Users, Suppression List, IP Access Rules, Send Connectors, Message Filters, Accepted Domains — sender and recipient — and all their detail pages) previously inherited only the deny-by-default `[Authorize]`, so a read-only **TenantViewer** could create/edit/delete through the UI (which calls services directly) even though the REST API blocks those mutations. All such pages now require `AdminFull`, and the per-tenant **Configuration** nav group is gated the same way (viewers see the monitoring groups only). The SMTP-user create flow also scopes its duplicate check and post-create lookup to the active tenant instead of relying on the ambient query filter (which is a no-op in host scope).
- **Open-relay protection is now enforced and cannot be disabled by configuration.** Relaying to an external (non-hosted) recipient requires either SMTP authentication or an *explicit, non-"any" allow-IP rule*; an empty configuration — or an overly-broad allow rule (`0.0.0.0/0`, `::/0`, or a near-"any" combination such as `0.0.0.0/1` + `128.0.0.0/1`) — no longer relays to the outside world. The check is hard-coded in `RelayMailboxFilter.CanDeliverToAsync` (an "any" allow rule still permits connections/inbound but never authorizes relaying). **Behaviour change:** a default/empty deployment previously accepted unauthenticated mail to any recipient (open relay); it now refuses external relay until you authenticate or add a specific allow-IP rule. Mail to domains the relay hosts (accepted recipient / backup-MX domains) is unaffected. The Health page gained a **Relay security** card showing the posture, and `SECURITY.md` plus a README "Responsible operation" section document safe operation.
- IP access rules are now evaluated per tenant on the shared SMTP listener. Once a message's owning tenant is resolved, only that tenant's rules plus the host/default-tenant baseline are evaluated (tenant rules first, then the baseline) — so one tenant's Allow/Deny rules can no longer permit or block another tenant's traffic. Previously all tenants' rules were evaluated as a single global list.
- The temporary-password change is now hard-enforced: an admin signed in with a one-time/reset password is redirected to the change-password page and cannot reach any other admin page until the password is changed (previously this was only a banner).
- Accepted recipient and sender domains are now globally unique across tenants — a domain may be claimed by only one tenant. Previously uniqueness was per-tenant, so two tenants could register the same domain and the sender/recipient → tenant attribution on the shared listener resolved ambiguously (first writer wins). Claiming a domain another tenant already holds is now rejected.
- Closed a cross-tenant aggregate leak in the monthly statistics view: the pre-aggregated `DailyStatistics` was global, so a tenant administrator's 30-day chart showed totals summed across all tenants. Daily statistics are now per tenant and the monthly query respects the tenant scope.
- Disabling a tenant now actually blocks it everywhere. A disabled tenant's admins can no longer sign in to the web admin (the password is still verified first, so a disabled-tenant account is not revealed by a wrong password), its API keys fail authentication, and inbound SMTP from/for that tenant is rejected. The enabled-tenant set is cached for the SMTP/API hot path and refreshed whenever a tenant is enabled, disabled, approved, or deleted. The default tenant can no longer be disabled, since it owns the baseline configuration.
- Closed the unauthenticated admin surface: the entire REST API and Blazor admin UI were previously reachable without credentials on `0.0.0.0:8025`, including user creation, configuration changes, and DKIM private-key generation.
- Closed a cross-tenant access path (IDOR): admin services/endpoints that loaded an entity by primary key used EF `Find`, which bypasses the tenant query filter. These now use filtered lookups, so a tenant administrator cannot read or modify another tenant's user, connector, domain route, DKIM domain, IP rule, message-filter rule, or queued message by guessing its id. (Blazor pages that open their own DI scope still need the ambient tenant propagated — addressed with the tenant-management UI.)
- Closed a cross-tenant DKIM signing gap: outbound mail is now signed with the **sending tenant's own** DKIM key, resolved by `(TenantId, sender domain)` from the database, so a tenant can never sign with another tenant's key — a non-default tenant with no key of its own simply signs nothing. The DB-backed DKIM Domains admin page now drives real signing; the file-based `Dkim` appsettings config is used as a fallback only for the default tenant (legacy single-tenant installs).
